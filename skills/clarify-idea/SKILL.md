# 아이디어 확인 (Clarify Idea)

목표: 계획이나 변경 요청의 결정 트리(decision tree)를 한 갈래씩 따라가며, 공통 이해(shared understanding)에 도달한다.

이 스킬은 가벼운 `grill-me`다. 파일을 쓰거나, PRD/TODO로 유도하거나, 기능 목록을 늘리는 스킬이 아니다. 질문은 짧게, 하나씩, 하지만 필요한 결정은 피하지 않는다.

## 진행 방식 (Workflow)

1. 현재 계획에서 아직 모호한 결정 하나를 고른다.
2. 질문 하나만 한다.
3. 질문마다 추천 답변과 그 이유를 함께 제시한다.
4. 답을 받으면 그 답을 전제로 다음 의존 결정으로 이동한다.
5. 코드나 문서를 보면 답할 수 있는 것은 묻지 말고 직접 확인한다.
6. 더 물어볼 결정이 없으면 짧게 "이제 충분히 명확합니다"라고 말하고 멈춘다.

## 질문 기준 (Question Targets)

아래를 체크리스트처럼 한 번에 묻지 않는다. 현재 결정에 필요한 것만 고른다.

- 누가 쓰는가?
- 어떤 순간에 여는가?
- 이번 변경에서 사용자는 무엇을 하는가?
- 결과로 무엇이 보여야 하는가?
- 어떤 데이터가 필요하고 저장해야 하는가?
- 사내망, 개인 로컬, 외부 공개 중 어디서 쓰는가?
- 성공했다고 어떻게 확인하는가?
- 이번에는 무엇을 하지 않는가?

## 질문 형식 (Question Format)

```markdown
질문: ...

추천: ...

이유: ...
```

추천은 기본값이다. 사용자가 다르게 답하면 사용자의 답을 따른다.

## 금지

- 질문 목록을 한 번에 던지지 않는다.
- 전체 앱 구조를 먼저 설계하지 않는다.
- 인증(auth), 결제(payment), 관리자(admin), 대시보드(dashboard), 설정(settings)을 기본값으로 넣지 않는다.
- "있으면 좋아 보이는 기능"을 이번 범위에 넣지 않는다.
- PRD, TODO, DESIGN.md, 구현 계획을 이 스킬 안에서 작성하지 않는다.
- 다음 스킬이나 다음 단계로 사용자를 밀지 않는다. 사용자가 요청하면 그때 진행한다.

## 멈추는 방식

더 물어볼 결정이 없으면 긴 산출물을 만들지 않는다. 짧게 현재 공통 이해를 확인하고 멈춘다.

```markdown
이제 충분히 명확합니다.
```
